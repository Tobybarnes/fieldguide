## For Week 1 (Access to Tools) - New Section

### Process Complexity and Personal Tools

Traditional software development tries to predict everything upfront: requirements, architecture, edge cases. It's Newtonian—plan, execute, control.

But that's not how you actually use technology. You encounter a problem, try a solution, notice what breaks, adjust. The tool evolves through use.

Complexity scientist Jean Boulton calls this "process complexity"—understanding the world as organic and in constant flow rather than as a machine to be predicted and controlled. Stability doesn't come from perfect planning. It emerges from ongoing, reflexive interaction.

That's exactly what building with Claude teaches you.

You describe → Claude generates → you test → you notice friction → you refine. The tool becomes useful not because you planned it perfectly, but because you responded to what emerged.

This is fundamentally different from:
- **Buying software** (someone else's prediction of what you need)
- **Traditional coding** (trying to anticipate everything upfront)
- **No-code platforms** (constrained by preset templates)

Field Guide embraces process complexity. Build fast, test in reality, adapt. The tool stabilizes through use, not through comprehensive planning.

## For Week 3 (The Loop) - Enhanced Opening

### Making Sense Through Making

Boulton writes about "making sense" versus jumping to quick solutions. Understanding patterns and processes is more important than having immediate answers.

The loop is a sense-making practice:

**Describe** → You articulate what you want clearly enough that someone (or something) else can build it. This is harder than it sounds. Vague ideas become explicit. You make sense of what you actually need.

**Generate** → Claude builds. You see your idea externalized.

**Test** → Reality talks back. The tool reveals what you missed in your description.

**Refine** → You adjust based on what emerged, not what you predicted.

Each cycle makes more sense. Not because you're getting smarter—because you're engaging with feedback from reality.

Traditional education teaches you to plan thoroughly before acting. This teaches you to act in order to learn. Small iterations, continuous adjustment, emergent understanding.

Boulton structures her book as 73 short, networked pieces—you can open it anywhere and find something useful. The practices in Field Guide work the same way. Each one is complete on its own, but they connect. You can build them in any order. The understanding emerges from doing them, not from reading about them.

## For Week 4 (The Loop) - Add This

### Reflexive Interaction

Boulton describes stability emerging from "reflexive interactions among diverse elements."

In traditional software development, you interact with requirements documents, architecture diagrams, test specs. All representations, all predictions.

In Field Guide, you interact with:
- **Your actual use** (not imagined use cases)
- **Real friction points** (not anticipated edge cases)  
- **What you notice** (not what metrics tell you)
- **The tool's behavior** (not the code itself)

This is reflexive—the tool shapes your understanding, you reshape the tool, repeat. Neither you nor the tool is fixed. Both adapt.

Claude accelerates this because it has no attachment to its previous solutions. It rebuilds without ego. You can try five different approaches in an hour, each one informing the next.

This is how complex systems actually work. Not through master plans, but through rapid cycles of interaction and adjustment.

## For Week 11 (Shipping) - New Section

### Fragility and Interconnection

Boulton emphasizes that we live in an "interconnected and fragile world" where small acts matter.

When you deploy a tool online, you're entering this interconnection. Your tool becomes part of:
- The web (connected to everything)
- Infrastructure (dependent on services staying online)
- Other people (if you share it)
- Maintenance (it will need care)

This isn't a warning—it's reality. And it informs how you approach shipping.

**Ship small.** Don't try to build a platform. Build one tool that does one thing.

**Ship for yourself first.** Know it works in your hands before sharing.

**Accept fragility.** Things break. Services change. URLs die. That's fine. Your tool doesn't need to last forever—it needs to work now.

**Choose appropriate complexity.** Don't add authentication if you're the only user. Don't scale if you don't need to. Match the tool's complexity to its actual use.

The traditional tech mindset is: build once, scale infinitely, last forever. But most tools don't need that. Most tools are temporary responses to current needs.

Accept this. It makes building simpler and shipping faster.

## For Week 12 (Refine) - Add This

### Patterns Over Solutions

Boulton distinguishes between "making sense" (understanding patterns) and "jumping to solutions."

When you revisit a tool to refine it, resist the urge to immediately add features. Instead:

1. **Use it consistently** for a week
2. **Notice patterns** in when it works and when it frustrates you
3. **Name the friction** specifically
4. **Understand why** it's friction (not just that it is)
5. **Then** ask Claude for a solution

Most people skip to step 5. They know something feels wrong and immediately want to fix it. But the fix only works if you understand the pattern.

Example:
- **Bad:** "This search is annoying. Add filters."
- **Good:** "I search for the same tags repeatedly. The search doesn't remember my last search. This means I type the same thing 5 times a day. Add a 'search history' dropdown that shows my last 5 searches."

The second version emerges from pattern recognition. You've watched yourself use the tool. You've noticed repetition. You've understood *why* it's frustrating.

This is slower upfront but produces better results. You're not feature-creeping—you're responding to observed patterns.

## For Week 13 (Make Your Own) - New Opening

### Small Acts in Complex Systems

Boulton writes about "small, compassionate acts" mattering in interconnected systems. Field Guide is about small tools, not grand platforms.

The traditional tech narrative is: identify a massive problem, build a venture-backed solution, scale to billions of users. Change the world.

But most real problems are small and personal. You need to track something specific. You want to organize something your way. You wish a certain interaction was faster.

These aren't billion-dollar problems. They're Tuesday afternoon problems.

And the tools that solve them don't need to be impressive. They need to work for you.

**This is the practice:**
- Notice a small friction
- Build a small tool
- Use it for a small while
- Adjust as needed
- Move on

No grand vision required. No startup formation. No pitch deck. Just: I need this, so I'll make it.

Boulton structures her book as 73 networked pieces because complex understanding doesn't come from linear progression—it emerges from connecting small insights.

Field Guide teaches the same way. Nine small practices. Each one complete. Together they add up to capability.

You don't need to build the perfect tool. You need to build many small ones. The capability emerges from the practice, not from any single creation.
